# COR-0045: Capability-denial audits omit worker-required resource identity

## Claim

Capability-denial audit events produced by `SandboxContext.RequireCapability(...)` are not schema-compatible with the worker audit validator. The producer writes a `PolicyDenied` event with a `CapabilityId` but no `ResourceId`, while `WorkerAuditValidator` only accepts `PolicyDenied` events whose resource is exactly `capability:{CapabilityId}`.

## Evidence

- `src/DotBoxd.Kernels/Sandbox/SandboxContext.cs:41` routes capability checks through `RequireCapability(...)`, and `src/DotBoxd.Kernels/Sandbox/SandboxContext.cs:45` writes a `PolicyDenied` audit when the policy does not grant the requested capability.
- That `SandboxAuditEvent` supplies `CapabilityId`, `ErrorCode`, and `Message`, but it does not set `ResourceId`, so the event cannot identify the denied capability resource in the shape expected elsewhere.
- `src/DotBoxd.Hosting/WorkerAuditValidator.cs:43` accepts `PolicyDenied` only when `auditEvent.ResourceId` equals `capability:{auditEvent.CapabilityId}`.
- `src/DotBoxd.Kernels/Sandbox/SandboxContext.cs:202` calls `RequireCapability(...)` from `ChargeBindingCall(...)` for any binding with a required capability, so an expired or otherwise missing grant during execution can produce this malformed denial event.

## Impact

A legitimate capability denial generated by the sandbox runtime can be rejected by worker audit validation because the event produced by core runtime code does not satisfy the host validator's own `PolicyDenied` schema. In isolated execution this can turn expected permission-denied evidence into worker-result validation failure; in in-process execution it also publishes an under-specified audit event that lacks the denied resource identity.

## Suggested acceptance test

Add a worker-audit or host execution regression that creates a context-generated capability denial, then asserts the resulting `PolicyDenied` event includes `ResourceId == "capability:<id>"` and passes `WorkerAuditValidator.Matches(...)` for the corresponding plan/options.

## Suggested fix direction

Set `ResourceId: $"capability:{capabilityId}"` on the `PolicyDenied` audit written by `SandboxContext.RequireCapability(...)`. Keep the validator strict so forged `PolicyDenied` events still need to bind the resource identity to the denied capability.

## Scope boundaries

This is separate from deterministic timestamp handling for synthesized binding failure audits: the issue here is the missing `ResourceId` contract between the runtime audit producer and worker audit validator.
